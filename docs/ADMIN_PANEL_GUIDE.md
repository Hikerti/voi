# Руководство по администрированию сайта Voitov Studio

Это руководство предназначено для менеджеров и редакторов, которым нужно управлять контентом через **Prisma Studio** и следить за входящими заявками. Здесь описаны шаги по запуску админ‑панели, правила редактирования данных, а также работа уведомлений через email.

## 1. Как запустить Prisma Studio

Prisma Studio — это визуальная админ‑панель базы данных. Она открывается как веб‑страница и позволяет изменять записи в таблицах без знания SQL.

1. **Откройте терминал** и перейдите в папку `backend` проекта:

   ```bash
   cd /path/to/voitov-studio/backend
   ```

2. **Запустите команду**:

   ```bash
   npm run prisma:studio
   ```

   Эта команда поднимает локальный сервер Prisma Studio. Если всё настроено по умолчанию, Studio станет доступна по адресу `http://localhost:5555`.

3. **Перейдите в браузер**. Введите в адресной строке `http://localhost:5555`. Если сайт развернут на сервере, админ‑панель проксируется через Nginx и обычно доступна по адресу `http://<IP‑сервера>:5555`. Конкретный адрес указан администратором проекта.

4. **Авторизация** не требуется: Prisma Studio открывается сразу. Рекомендуется ограничивать доступ к порту 5555 через файрвол или Nginx, чтобы админ‑панель не была открыта для всех в интернете.

> **Совет.** При первом запуске Studio может предложить применить миграции. На боевом сервере это делает разработчик. Для менеджеров достаточно читать/изменять данные.

## 2. Общие правила редактирования данных

Перед началом работы важно понимать некоторые общие правила, которые применяются ко всем моделям:

- **`slug`** — это часть ссылки (URL) записи. Она используется в адресах страниц и в API. Пишите только латинскими буквами, в нижнем регистре, слова разделяйте дефисами: например `landing-page` или `seo-start`. Изменение `slug` меняет URL страницы, поэтому делайте это только на этапе создания записи.
- **`sortOrder`** — номер сортировки. Чем меньше число, тем выше запись в списке на сайте. Для равных чисел сортировка определяется по дате.
- **`status`** — статус публикации. Допустимые значения:
  - `DRAFT` — черновик, не показывается на сайте.
  - `PUBLISHED` — опубликовано и видно посетителям.
  - `HIDDEN` — скрыто, но остаётся в базе.
- **SEO‑поля** (`seoTitle`, `seoDescription`, `seoKeywords`, `h1`) заполняйте для улучшения выдачи в поисковых системах. Поле `h1` задаёт заголовок страницы, если он должен отличаться от основного названия.
- **`createdAt`** и **`updatedAt`** выставляются автоматически. Менять их вручную не нужно.

### Как редактировать запись

1. В левой панели выберите нужную таблицу. Таблицы перечислены ниже.
2. Кликните на строку, чтобы открыть форму редактирования. Все поля отображаются с именами из модели.
3. Заполните или измените поля. Для текстовых блоков (описания, контент) используйте обычный текст; переносы строк сохраняются.
4. После изменений нажмите **Save 1 change** (кнопка появляется в верхнем правом углу). Это сохранит запись в базе данных.
5. Проверьте страницу на сайте, чтобы убедиться, что всё отображается корректно.

## 3. Описание основных таблиц

Ниже приведены основные модели и краткое описание их полей. Они соответствуют блокам на сайте. Полные определения доступны разработчикам в файле `PRISMA_STUDIO_GUIDE.md`.

### SiteSettings — общие настройки сайта

* **Блок:** контакты, соцсети, бренд и SEO по умолчанию.
* **Как использовать:** таблица содержит **одну запись** с `id = 1`. Здесь редактируются название бренда (`brandName`), слоган (`tagline`), контактный email (`email`), телефон (`phone`), график работы (`workHours`), адрес (`address`), ссылки на соцсети (`telegramUrl`, `vkUrl`, `youtubeUrl`), а также SEO‑поля по умолчанию и картинка для превью (`ogImage`). Изменяйте эти поля по мере необходимости.

### ServiceCategory — категории услуг

* **Блок:** объединяет услуги в группы (дизайн, digital, маркетинг, стратегия).
* **Поле `title`** — название категории, которое видят пользователи.
* **Поле `slug`** — внутренний ключ; публичных страниц категорий нет.
* **`services`** — связь с услугами; отображается автоматически.

### Service — услуги

* **Блок:** карточки услуг и страницы `/services/{slug}`.
* **Основные поля:** `title`, `slug`, `summary` (краткое описание), `description` (полное описание), `price` и `pricePrefix`, `image` (путь к изображению в папке `/public/images`), `seoText` (SEO‑текст) и связи с пакетами и похожими услугами.
* **Категория (`categoryId`)** может быть пустой, если услуга не относится ни к одной категории.

### PricePackage — цены/тарифы

* **Блок:** таблица тарифов на главной и в карточках услуг.
* **Использование:** для каждого пакета указывайте `title`, `price` и список особенностей в поле `features` (перечисляйте через `;` или в формате JSON, как согласовано с разработчиками). Поле `serviceId` задаёт услугу, к которой относится пакет; оставьте пустым для общих тарифов.

### Advantage — преимущества студии

* **Блок:** «Преимущества»/«Почему мы» перед футером.
* **Поля:** `title`, `slug`, `summary` (текст преимущества), `icon` (ключ иконки) и стандартные поля сортировки/статуса.

### WorkStage — этапы процесса

* **Блок:** «Как мы работаем».
* **Поля:** `title`, `slug`, `summary`, `stepNumber` (номер шага в процессе), `sortOrder` и `status`.

### FAQItem — вопросы и ответы

* **Блок:** раздел FAQ на главной и странице `/faq`.
* **Поля:** `question`, `slug`, `answer`, `category` (если нужно группировать), `sortOrder`, `status` и SEO‑поля.

### Review — отзывы

* **Блок:** отзывы на главной и страница `/reviews`.
* **Поля:** `authorName`, `company`, `role`, `text` (сам отзыв), `rating` (оценка), `sourceUrl` (ссылка на оригинал), `avatar` (ссылка на аватар автора). Поле `sortOrder` определяет порядок, `status` — видимость.

### NewsPost — новости

* **Блок:** лента новостей и страницы `/news/{slug}`.
* **Поля:** `title`, `slug`, `excerpt` (анонс), `content` (полный текст), `coverImage`, `publishedAt` (дата публикации), `sortOrder`, `status` и SEO‑поля.

### Article — статьи/блог

* **Блок:** блог и страницы `/blog/{slug}`.
* **Поля:** аналогичны `NewsPost`: `title`, `slug`, `excerpt`, `content`, `coverImage`, `publishedAt`, `sortOrder`, `status` и SEO‑поля.

### Project — портфолио

* **Блок:** раздел «Портфолио» и страницы `/portfolio/{slug}`.
* **Поля:** `title`, `slug`, `client` (название клиента), `industry` (ниша), `summary` (краткое описание), `description` (полное описание), `coverImage` (основная картинка), `gallery` (список изображений; удобнее хранить через `;` или JSON), `projectUrl` (ссылка на живой сайт), `sortOrder`, `status` и SEO‑поля.

### LeadRequest — заявки/обращения

* **Блок:** все формы на сайте (контактная форма, обратный звонок, отзыв, вопрос и т.д.).
* **Просмотр:** записи здесь появляются автоматически, когда посетители оставляют заявки. Для каждой заявки фиксируются тип (`kind`: `CONTACT`, `CALLBACK`, `REVIEW`, `QUESTION`), статус (`status`: `NEW`, `IN_PROGRESS`, `DONE`, `SPAM`), контактные данные (`name`, `phone`, `email`), сообщение (`message`), источник (`source`), URL страницы (`pageUrl`), IP и user‑agent.
* **Как работать:** менеджеры обычно просматривают заявки и изменяют только поле `status`. Например:
  - `NEW` — новая заявка, никто ещё не обработал.
  - `IN_PROGRESS` — заявка в работе: менеджер связался с клиентом.
  - `DONE` — заявка обработана (заказ закрыт, ответ отправлен и т.д.).
  - `SPAM` — нежелательная или ошибочная заявка.

## 4. Мониторинг заявок (LeadRequest)

Для контроля входящих обращений выполните следующие действия:

1. Откройте Prisma Studio и выберите таблицу **LeadRequest**.
2. В списке увидите все заявки. По умолчанию они упорядочены от новых к старым. Поля `kind` и `status` помогут быстро понять тип заявки и её текущее состояние.
3. Чтобы изменить статус заявки, кликните на нужную строку, выберите новое значение в поле `status` и нажмите **Save 1 change**. Это позволяет отмечать, что вы взяли заявку в работу или закрыли её.
4. Дополнительные поля (`name`, `phone`, `email`, `message`, `pageUrl`) показывают, откуда пришла заявка и что написал клиент. Эти данные не нужно редактировать.
5. Если заметили, что телефон не указан — это нормально. В последних версиях формы телефон не является обязательным полем (см. миграцию, делающую `phone` необязательным), но хотя бы один контакт (телефон или email) должен присутствовать.

> **Важно.** Не удаляйте заявки и не меняйте другие поля кроме `status`. Поле `source` используется аналитикой для понимания, с какого блока сайта пришёл пользователь.

## 5. Интеграция с почтой (уведомления о заявках)

Форма на сайте не только сохраняет заявку в базу данных, но и отправляет уведомление на указанный email. Процесс работает следующим образом:

1. Пользователь заполняет форму (контакт, обратный звонок, отзыв или вопрос) на сайте. Данные отправляются на внутренний маршрут `/contact-handler` на фронтенде.
2. Файл `lib/contact-handler.ts` (Next.js Route Handler) валидирует введённые данные: проверяет корректность телефона и email, наличие согласия на обработку данных, ограничивает частоту отправки. Если всё в порядке, функция отправляет запрос на API бэкенда (`/api/leads/contact`, `/leads/callback` и т.д.), сохраняя заявку в таблице `LeadRequest` в базе данных【77†L96-L143】.
3. После успешного создания заявки, `contact-handler.ts` формирует уведомление и отправляет его через сервис **Resend** (https://resend.com). Функция `sendEmail` собирает текст письма (ID заявки, имя, телефон, email, сообщение и источник) и выполняет POST‑запрос к API Resend【77†L150-L200】.
4. Чтобы email‑уведомления работали, необходимо настроить переменные окружения в файле `.env.local` (на стороне фронтенда):

   - `CONTACT_EMAIL_TO` — адрес (или список адресов), на который будут приходить уведомления о заявках.
   - `CONTACT_EMAIL_FROM` — имя/адрес отправителя, например `"Voitov Studio <site@voitov.ru>"`.
   - `RESEND_API_KEY` — секретный ключ от сервиса Resend【78†L26-L37】. Этот ключ хранится только на сервере и **не** выкладывается в репозиторий.

5. Если Resend не настроен (переменные не заполнены), заявка всё равно сохранится в базе, но уведомление на почту отправлено не будет. Фронтенд в ответ покажет сообщение об успешной отправке без email‑уведомления【77†L150-L159】.

6. Обрабатывая заявки в Prisma Studio, менеджер может связаться с клиентом, используя указанные телефон или email. Сама почтовая интеграция на бэкенде не требует дополнительных действий: всё происходит в Next.js‑обработчике.

### Настройка почтовой интеграции при развертывании

При деплойменте разработчики добавляют переменные в файл `.env.local` проекта (фронтенд) и перезапускают сборку. Минимальный набор переменных выглядит так【78†L26-L39】:

```env
CONTACT_EMAIL_TO=example@domain.com        # получатель уведомлений
CONTACT_EMAIL_FROM="Voitov Studio <noreply@voitov.ru>"  # от кого приходит письмо
RESEND_API_KEY=your-resend-api-key         # секретный ключ Resend
```

После установки или изменения этих значений нужно пересобрать frontend (`npm run build`) и перезапустить сервер, чтобы новые настройки вступили в силу.

## 6. Часто задаваемые вопросы

- **Можно ли добавлять изображения через Prisma Studio?**

  В большинстве случаев изображения предварительно загружаются в папку `public/images` с помощью Git или через систему управления файлами. В Prisma Studio вы указываете только относительный путь (например `/images/banner.jpg`). Если нужно загрузить новое изображение, обратитесь к разработчику.

- **Что делать, если Prisma Studio не запускается?**

  Проверьте, что вы находитесь в папке `backend`, и что установлены зависимости (`npm install`). Убедитесь, что порт 5555 свободен. Если на сервере порт закрыт, обратитесь к администратору для настройки Nginx/файрвола.

- **Как ограничить доступ к Prisma Studio в production?**

  В Nginx конфигурации предусмотрен прокси на порт 5555 только по конкретному IP‑адресу сервера【78†L119-L124】. Убедитесь, что этот IP указан верно. Дополнительно можно настраивать базовую HTTP‑авторизацию или VPN.

- **Можно ли редактировать заявки?**

  Не рекомендуется изменять содержимое заявок. Меняйте только поле `status` для отслеживания процесса обработки. Если нужно удалить спам‑заявку, можно присвоить ей статус `SPAM`.

## 7. Заключение

Prisma Studio — удобный инструмент для управления контентом и заявками без участия разработчиков. Соблюдайте общие правила (корректный `slug`, статус публикации, порядок сортировки) и не забывайте проверять итоговый вид страниц на сайте. Для работы уведомлений о заявках обязательно настройте переменные `CONTACT_EMAIL_TO`, `CONTACT_EMAIL_FROM` и `RESEND_API_KEY`. Если возникают вопросы, обращайтесь к разработчикам.
